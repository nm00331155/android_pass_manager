# VS Code / GitHub Copilot 向け 実装指示プロンプト
## SecureVault: メールOTP補助導線 + クリップボード復帰自動入力を 1 回で実装する

あなたは、このリポジトリ `android_pass_manager` の既存実装を尊重しながら、**既存OTP導線を壊さずに拡張する**慎重で優秀な Android エンジニアです。  
対象アプリは **SecureVault** です。

この作業では、**メール本文を自動解析する機能は実装しません**。  
代わりに、OTP入力欄で SMS 以外のケースにも対応しやすくするため、**ユーザー補助型のメールOTP導線**を追加してください。

---

## 今回の目的

OTP 入力欄を検知したとき、SMSだけでなく**メールで届くワンタイムコード**にも対応しやすくするため、以下の補助フローを実装してください。

### 実現したいフロー
1. OTP 入力欄を検知する
2. OTP 補助画面 / 待機UIを表示する
3. その画面に **「メールアプリを開く」** 導線を追加する
4. ユーザーがメールアプリを開いて、自分でワンタイムコードをコピーする
5. SecureVault に戻る
6. 戻ってきたタイミングで clipboard を再確認する
7. OTP らしいコードが clipboard にあれば、それを利用して自動入力できるようにする

---

## この実装の前提

- **メール本文の自動読取はしない**
- **メールアプリそのものを解析しない**
- あくまで「メールを開く導線」と「戻ってきた後の clipboard 補助」を実装する
- 既存の
  - SMS OTP
  - 通知 OTP
  - Clipboard OTP
  - recent OTP
  - Autofill OTP dataset
  の流れを壊さないこと
- 今回の実装は**補助機能**であり、SMS OTP の代替本命ではない

---

## 最重要方針

- **新規大機能追加ではなく、既存OTP導線の自然な拡張として実装する**
- 既存の OTP フローを壊さない
- UI / 導線は分かりやすく、過剰に複雑にしない
- まず実装に必要な既存箇所を調査し、最小差分で実装する
- **INTERNET 権限は追加しない**
- メール本文読取や非公開API利用はしない
- Android の一般的な Intent / chooser / clipboard / lifecycle の範囲で実装する
- docs 更新を忘れない

## 今回実装してほしい内容

---

### 1. OTP待機UIに「メールアプリを開く」導線を追加

#### 目的
メールで OTP を受け取るケースでも、ユーザーが迷わずコード取得に移れるようにする。

#### 要件
- OTP待機画面（既存の `SmsOtpActivity` または関連UI）に、以下のような補助説明とボタンを追加する
  - 例:  
    - 「メールでコードを受け取った場合は、メールアプリを開いてコードをコピーしてください」
    - 「SecureVault に戻ると自動で検出します」
- ボタン文言は、以下のどちらかで分かりやすい方を採用
  - `メールアプリを開く`
  - `コードを受け取るアプリを開く`
- 初回実装では **メールアプリを開く** を優先してよい

#### 実装方針
- 既存 OTP 待機UI の自然な拡張にする
- SMS待機を邪魔しない
- ボタンを押したら chooser を表示できるようにする

---

### 2. メールアプリ chooser を起動する

#### 目的
ユーザーが Gmail / Outlook / Yahoo メールなど、自分の使うメールアプリを選んで開けるようにする。

#### 要件
- Android 標準的な Intent を使って、メール系アプリを開く導線を実装する
- 端末依存を考慮し、**完全なメールアプリ限定列挙が無理でもベストエフォートでよい**
- 失敗した場合はクラッシュさせず、ユーザーに分かる形でフォールバックする

#### 実装方針
以下の優先順で検討してください。

1. `Intent.ACTION_MAIN` + `CATEGORY_APP_EMAIL`
2. chooser intent
3. resolve できなければ一般的なアプリ選択 fallback またはメッセージ表示

#### 注意
- Gmail 固定にしない
- 特定ベンダや特定アプリ前提にしない
- 直接本文表示や特定受信メール表示は狙わない

---

### 3. SecureVault 復帰時に clipboard を再確認する

#### 目的
ユーザーがメールアプリで OTP をコピーして戻ってきた時、既存 clipboard OTP 機構で自然に拾えるようにする。

#### 要件
- OTP待機UIが `onResume` / lifecycle resume 時に clipboard を再評価できること
- 既存の `ClipboardOtpDetector` / `OtpManager` をできるだけ再利用すること
- OTPらしい新しい clipboard 内容があれば、既存フローに流せるようにすること

#### 実装方針
- 既存 clipboard 監視が常時監視型なら、そのまま使ってよい
- ただし「メールから戻った直後に再チェックされる」ことを明確に担保する
- 必要なら resume 時に明示的な再検査メソッドを追加してよい

---

### 4. 古い clipboard 値の誤検出を減らす

#### 目的
まったく関係ない古いコピー内容を OTP として誤採用しないようにする。

#### 要件
以下のいずれか、または複数を組み合わせて対策してください。

- OTP待機開始後に新しくコピーされたものを優先する
- recent OTP の有効期限を使う
- clipboard から拾うOTPは一定時間内のみ有効にする
- OTP欄検知中だけ clipboard 補助を強める
- 明らかに古い値は使わない

#### 実装方針
- 今ある `OtpManager.getRecentOtp(...)` の仕組みや timestamp を活用する
- 無理に大改修せず、最小差分で精度を上げる

---

### 5. OTP復帰後のUXを自然にする

#### 目的
ユーザーが「メールを開く → コピー → 戻る」したときに、何が起こるか分かりやすくする。

#### 要件
以下のどちらか、または既存フローに沿った自然な案を採用してください。

- clipboard の OTP を検知したら、そのまま autofill dataset として使う
- または「コピーしたコードを入力しますか？」と確認してから使う

#### 優先方針
- 既存 OTP 補助フローに沿うこと
- UI を増やしすぎないこと
- 初回実装では既存 recent OTP / autofill dataset 連携に自然に乗る方式を優先する

---

### 6. 設定や仕様文言も必要なら整理する

#### 目的
ユーザーに「メールOTPは自動読取ではなく補助導線である」ことを誤解なく伝える。

#### 要件
必要なら以下を更新する。

- OTP待機画面文言
- 設定画面の補助説明
- docs/implemented.md
- docs/pending.md
- 必要なら README の機能説明

#### 文言方針
- メール本文を自動取得するとは書かない
- 「メールアプリを開いてコードをコピーし、戻ると検出できます」という補助表現にする
- Android制約に沿った現実的な説明にする

## 実装時の具体条件

### 既存機能を壊さないこと
- SMS OTP はそのまま維持
- 通知 OTP はそのまま維持
- Clipboard OTP はそのまま維持
- Autofill OTP dataset の流れは壊さない

### 追加してよいもの
- メールアプリ起動 helper
- lifecycle resume での clipboard 再チェック
- OTP補助説明文言
- 必要最小限の state / flag

### 追加しないもの
- メール本文解析
- Gmail 専用処理
- Outlook 専用処理
- 非公開 API
- クラウド連携
- 大規模 OTP アーキテクチャ変更

## 期待する完成イメージ

### ユーザー体験
- OTP 欄で SecureVault の OTP 補助が出る
