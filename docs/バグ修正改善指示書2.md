根本原因の分析
すべての問題は一つの設計上の問題に帰結します。

MasterKeyManager で Android Keystore のキーを生成する際に:

Copy.setUserAuthenticationRequired(true)
.setUserAuthenticationParameters(AUTH_TIMEOUT_SECONDS, ...)  // 300秒 = 5分
これにより、Keystore のキーは 生体認証後 300秒間だけ使用可能 になります。つまり:

問題1（読み込みが遅い / フリーズ）: CredentialRepositoryImpl.getAll() が呼ばれると、DBから取得した全レコードの toDomainOrNull() で 1件ごとに cryptoEngine.getCipherForDecryption(iv) を呼びます。これは内部で masterKeyManager.getOrCreateMasterKey() → Keystore アクセスを 1件あたり3回（username, password, notes）行います。50件なら 150回の Keystore アクセス。さらに認証タイムアウト後はキーが使えなくなるので、各回で内部的にブロックや例外が発生する可能性があります。

問題2（パスワードを呼び出せない / データが消える）: 認証から300秒（5分）が経過すると、setUserAuthenticationRequired(true) によりキーが UserNotAuthenticatedException をスローします。その結果:

getCipherForDecryption → 例外 → toDomainOrNull() が null を返す → 全データが「消える」
getCipherForEncryption → KeyPermanentlyInvalidatedException と混同して recreateMasterKey() が呼ばれ、新しいキーが生成されて旧データが永久に復号不能になる
修正指示書（Phase 6 – 第3回）
修正の方針
setUserAuthenticationRequired(true) を 削除 する。代わりにアプリレベルの認証（生体認証画面 + AutoLockManager）でセキュリティを担保する。これが正しい設計で、Bitwarden や他のパスワードマネージャーも同じアプローチを取っている。

Keystore のキーに userAuthenticationRequired をつけるのは、「キーの使用時に毎回認証を要求する」ためのもので、パスワードマネージャーのように頻繁にキーを使うアプリには不向き。アプリレベルで認証画面を設けて AutoLock で制御すれば十分安全。

修正2: CryptoEngine — recreateMasterKey の危険な呼び出しを修正
ファイル: data/crypto/CryptoEngine.kt

getCipherForEncryption で KeyPermanentlyInvalidatedException だけでなく UserNotAuthenticatedException もキャッチする。しかし修正1で userAuthenticationRequired を削除すれば、この例外自体が発生しなくなる。念のため安全なフォールバックにする:

Copyfun getCipherForEncryption(): Cipher {
    val cipher = Cipher.getInstance(TRANSFORMATION)
    val key = masterKeyManager.getOrCreateMasterKey()
    cipher.init(Cipher.ENCRYPT_MODE, key)
    return cipher
}
try-catch の KeyPermanentlyInvalidatedException ブロックを削除する。userAuthenticationRequired がなければこの例外は発生しない。万が一発生した場合は、上位の runCatching でエラー表示される方が、黙ってキーを再作成してデータをロストするより遥かに安全。

同様に getCipherForDecryption:

Copyfun getCipherForDecryption(iv: ByteArray): Cipher {
    val cipher = Cipher.getInstance(TRANSFORMATION)
    val spec = GCMParameterSpec(GCM_TAG_LENGTH, iv)
    cipher.init(Cipher.DECRYPT_MODE, masterKeyManager.getOrCreateMasterKey(), spec)
    return cipher
}
修正3: CredentialRepositoryImpl — 復号の効率化
ファイル: data/repository/CredentialRepositoryImpl.kt

現在、1件の復号で getCipherForDecryption を3回呼んでいる（username, password, notes）。修正1でタイムアウト問題は解消されるが、パフォーマンス改善としてKeystore アクセスを減らす:

方法: 各フィールドで別々のIVを使っているため、フィールドごとに Cipher を作成する必要がある（AES-GCM の仕様上、IVが異なれば Cipher インスタンスも別にする必要がある）。ただし getOrCreateMasterKey() のキー取得自体は1回だけにできる。

CredentialRepositoryImpl にキーのキャッシュは不要（MasterKeyManager が @Singleton で同じ SecretKey 参照を返すため、Keystore アクセスは初回のみ）。修正1で userAuthenticationRequired を外せば、キー取得は高速になる。

ただし、getAll() のFlowが map 内でN件を直列復号するのは Dispatchers.Main で実行されUIがフリーズする。flowOn(Dispatchers.Default) を追加:

Copyimport kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.flow.flowOn

override fun getAll(): Flow<List<Credential>> {
    return credentialDao.getAll()
        .map { list -> list.mapNotNull { it.toDomainOrNull() } }
        .flowOn(Dispatchers.Default)

修正4: DbKeyManager — CryptoEngine との循環的な問題を防止
DbKeyManager.getOrCreatePassphrase() は cryptoEngine.getCipherForEncryption() を呼ぶ。修正1で userAuthenticationRequired を削除すれば問題なく動作する。変更不要。

修正後の手順（重要）
修正を適用した後、必ず以下の手順を実行する:

端末からアプリをアンインストール（adb uninstall com.securevault.app または端末の設定からアンインストール）
./gradlew :app:assembleDebug でビルド
再インストール
生体認証でロック解除
Chrome/Brave の CSV をインポート
ホーム画面でデータが即座に表示されることを確認
アプリをバックグラウンドに送り、5分以上待ってから戻る
再認証後、データがすべて残っていることを確認
アンインストールが必要な理由: 旧キー（userAuthenticationRequired=true）で作成された Keystore エントリと、それで暗号化された DataStore のシードが残っていると、新しいコードで復号できない。クリーンインストールが必要。

なぜこれで全問題が解決するか
「読み込みが10秒以上かかる」: userAuthenticationRequired により各 Keystore アクセスで認証チェックが入り、タイムアウト付近では内部的にブロックや再試行が発生。これがなくなれば Keystore アクセスはミリ秒単位。さらに flowOn(Dispatchers.Default) でUIスレッドをブロックしない。

「パスワードを呼び出せない」: 認証から5分経過後に UserNotAuthenticatedException が発生し、toDomainOrNull() で全件が null になっていた。userAuthenticationRequired を外すことでこの例外が発生しなくなる。

「インポート後にデータが消える」: 同上。認証タイムアウト後の復号失敗、または getCipherForEncryption の catch 内で recreateMasterKey() が呼ばれてキーが再生成され、旧データが復号不能になっていた。両方解消される。

「インポートが遅い」: saveAll のバッチ化は前回対応済み。加えて userAuthenticationRequired の除去で暗号化時の Keystore アクセスが高速化。50件のインポートが1〜3秒に改善されるはず。

この3ファイルの修正 + アンインストール/再インストールで、報告されたすべての問題が解消します。変更量は少ないですが効果は劇的です。一括でCopilotに渡してください。
